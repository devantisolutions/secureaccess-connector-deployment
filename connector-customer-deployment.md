# Connector Customer Deployment

The Connector is installed at the customer/private site and opens an outbound
WSS tunnel to Devanti Gateway. New dynamic connectors start with a one-time
onboarding token. During registration, the Control Plane may issue a
connector-specific mTLS client certificate bundle that the connector stores
locally for future tunnel authentication.

Do not bake, copy, or reuse another connector's `client.crt` or `client.key`.
Those files are connector identity material. Reusing them makes the new
connector authenticate as the original connector and can make both connectors
show degraded or offline.

## Customer Flow

1. Admin creates a connector record with a meaningful name.
2. Admin generates an onboarding token for that connector.
3. Customer deploys Connector OVA or container.
4. Customer configures:
   - static IP
   - subnet/CIDR
   - default gateway
   - DNS
   - `GATEWAY_URL=https://access.devantisolutions.com:9443`
   - connector token
5. Connector registers with Control Plane and resolves org/connector from the token.
6. If mTLS issuance is enabled, Control Plane generates a fresh client cert/key
   for that exact org and connector ID, returns it in the registration response,
   and the connector writes it under `/etc/devanti/connector/certs`.
7. Connector opens WSS tunnel to Gateway using the issued connector certificate.
8. Admin verifies status online and active tunnel present.

## Connector Certificate Lifecycle

There are four different certificate concepts:

| File or cert | Generated by | Purpose | Should be in OVA? |
|---|---|---|---|
| Public TLS cert for `secureaccess.devantisolutions.com` / `access.devantisolutions.com` | Let's Encrypt or enterprise CA on the cloud gateway | Browser/API/Gateway HTTPS identity | No |
| Gateway connector CA, usually cloud `certs/ca.crt` and `certs/ca.key` | Devanti control-plane deployment | Signs connector client certificates | No |
| Connector client cert/key, `/etc/devanti/connector/certs/client.crt` and `client.key` | Control Plane during token registration | Proves one connector identity during mTLS tunnel auth | No |
| Local UI cert/key, `/etc/devanti/connector/certs/ui/ui.crt` and `ui.key` | Connector appliance | Self-signed HTTPS for the local management UI | Yes, or generated on first boot |

Expected first registration:

1. Admin creates connector record, for example `c_home-lab_6701nq`.
2. Admin generates a one-time token from that connector row.
3. Connector starts with `CONNECTOR_MTLS_ENABLED=false` and no client cert/key.
4. Connector POSTs `connector_id`, `org_id`, and token to Control Plane.
5. Control Plane validates the token, persists the connector as online, signs a
   new connector-specific client certificate, stores its serial/fingerprint, and
   returns `cert_pem`, `key_pem`, and `ca_pem`.
6. Connector writes:
   - `/etc/devanti/connector/certs/client.crt`
   - `/etc/devanti/connector/certs/client.key`
   - `/etc/devanti/connector/certs/ca.crt`
7. Connector updates local config/env so mTLS is enabled for later starts.
8. Connector connects to `wss://access.devantisolutions.com:9443/api/v1/connectors/ws`.
9. Gateway verifies the client certificate and asks Control Plane to confirm the
   cert serial/fingerprint belongs to the same org and connector ID.

The onboarding token is single-use. After a successful registration, replacing
the token alone will not change connector identity; the connector identity is
the issued client certificate. To move an appliance to a different connector
record, stop the connector, move aside `client.crt` and `client.key`, configure
the new connector ID and a fresh token from that connector row, then start it.

## Firewall Rule

Required:

```text
TCP 9443 outbound to access.devantisolutions.com
TCP 443 outbound to registry-1.docker.io
TCP 443 outbound to auth.docker.io
TCP 443 outbound to production.cloudflare.docker.com when Docker Hub uses Cloudflare layer delivery
```

No inbound rule from Devanti is required for normal connector operation. The connector local HTTPS UI listens on TCP 443 only for the customer management network.

## First-Boot Runtime Config Example

```yaml
connector_id: "c_bangalore-lab-new_wcg4nq"
gateway_url: "https://access.devantisolutions.com:9443"
gateway_ws_url: "wss://access.devantisolutions.com:9443/api/v1/connectors/ws"
log_level: "info"
reconnect_interval: "5s"
mtls:
  enabled: false
tls:
  mode: "strict"
```

Environment file example:

```env
CONNECTOR_CONFIG_FILE=/etc/devanti/connector/config.yaml
CONNECTOR_MTLS_ENABLED=false
CONNECTOR_TOKEN=<connector-token>  
GATEWAY_URL=https://access.devantisolutions.com:9443
GATEWAY_WS_URL=wss://access.devantisolutions.com:9443/api/v1/connectors/ws
CONTROL_PLANE_URL=https://access.devantisolutions.com:9443
CONNECTOR_ID=c_bangalore-lab-new_wcg4nq
CONNECTOR_ORG_ID=1
CONNECTOR_TLS_MODE=strict
CONNECTOR_SERVER_NAME=access.devantisolutions.com
```

Use `CONNECTOR_MTLS_ENABLED=false` only for first registration when no
connector-specific client cert/key exists yet. After successful enrollment, the
connector writes the issued cert files and enables mTLS for the tunnel.

Do not point `CONTROL_PLANE_URL` at the admin/user portal hostname for connector
registration. Use the connector endpoint, normally the same value as
`GATEWAY_URL`: `https://access.devantisolutions.com:9443`.

## Public Docker Hub Image

Customer connector deployments use the public Docker Hub image:

```bash
docker pull devantisolutions/secureaccess-connector:latest
```

Minimal Docker run example:

```bash
docker run -d \
  --name devanti-connector \
  --restart unless-stopped \
  --env-file /etc/devanti/connector/connector.env \
  -p 443:443 \
  -v /etc/devanti/connector:/etc/devanti/connector:rw \
  -v /etc/ssl/certs:/etc/ssl/certs:ro \
  devantisolutions/secureaccess-connector:latest
```

For a full appliance-style install, the host must also have the Devanti helper
scripts and mount them into the container:

```text
/usr/local/sbin/devanti-connector-setup
/usr/local/sbin/devanti-connector-update
```

These scripts are not downloaded automatically by `docker run`. They are
installed by the OVA build or by the customer-side installer from the connector
deployment repository. Without `devanti-connector-update` mounted into the
container, the local UI will show `Updater missing`.

Required appliance mounts:

```text
-v /etc/devanti/connector:/etc/devanti/connector:rw
-v /etc/ssl/certs:/etc/ssl/certs:ro
-v /etc/netplan:/etc/netplan:rw
-v /usr/local/sbin/devanti-connector-update:/usr/local/sbin/devanti-connector-update:ro
-v /var/run/docker.sock:/var/run/docker.sock
```

The local UI is served by the connector process itself over HTTPS on TCP 443.
It calls local endpoints such as `/api/appliance`, `/api/status`,
`/api/logs`, and `/api/update/check`; those endpoints read/write local files
under `/etc/devanti/connector` and run the host-mounted update command.


## Local Connector UI Repository Fields

In the HTTPS connector UI, public customer deployments should use:

```text
Repository server: docker.io
Connector image: devantisolutions/secureaccess-connector:latest
```

Use this exact image value:

```text
devantisolutions/secureaccess-connector:latest
```

then the VM is probably still using an older systemd unit that hardcoded the lab/private image. Install the current `devanti-connector.service`, run `sudo systemctl daemon-reload`, and restart `devanti-connector`. The current unit honors `CONNECTOR_IMAGE`, normalizes legacy defaults, and starts the container with `devantisolutions/secureaccess-connector:latest` unless a custom image is intentionally configured.

## Installer Template

This repository includes a customer-side installer template:

```bash
sudo ./scripts/install-connector.sh \
  --image devantisolutions/secureaccess-connector:latest \
  --gateway-url https://access.devantisolutions.com:9443 \
  --connector-id c_bangalore-lab-new_wcg4nq \
  --token '<connector-token>' \
  --config-dir /etc/devanti/connector \
  --systemd
```

The script writes `config.yaml`, `connector.env`, and optionally a systemd
service. It prints the generated token-only `ExecStart` so the operator can
review it before enabling the service.

Enable after review:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now devanti-connector
sudo journalctl -u devanti-connector -f
```

Use the Connector OVA setup flow when deploying as a virtual appliance. The OVA
console asks for network settings, Gateway URL, and connector token.

## Validation

On connector:

```bash
systemctl status devanti-connector
journalctl -u devanti-connector -f
curl -vk https://secureaccess.devantisolutions.com/access/health
curl -vk https://access.devantisolutions.com:9443
docker pull devantisolutions/secureaccess-connector:latest
```

Expected logs:

- connector registered
- connector websocket auth success
- connector tunnel connected

In Admin Portal:

- connector heartbeat status online
- last_seen updates
- gateway tunnel active
- mapped HTTP/HTTPS/RDP/SSH resources launch through that connector
